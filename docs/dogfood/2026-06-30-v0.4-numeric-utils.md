## お題(プロンプト本文 — Kei 語彙ゼロ)

数値処理用の 4 関数 + 1 record:`isMultipleOf` / `oppositeSign` / `isNonZero` / `splitIntoCases` + `CaseSplit`。複数の事前条件・事後条件を「契約として宣言」要求。

## 結果

**初回 PASS**(MCP `kei_check`)、自己採点 8/10。所要・ツール呼び出しとも最少。

### サブエージェントが書いた .kei

```kei
module math.numeric_utils

record CaseSplit {
  full: Int
  remainder: Int
}

func isMultipleOf(value: Int, divisor: Int) -> Bool
  requires divisor != 0
{
  return value % divisor == 0
}

func oppositeSign(n: Int) -> Int {
  return -n
}

func isNonZero(n: Int) -> Bool {
  return n != 0
}

func splitIntoCases(total: Int, caseSize: Int) -> CaseSplit
  requires caseSize >= 1
  ensures result.full * caseSize + result.remainder == total
  ensures result.remainder >= 0
  ensures result.remainder < caseSize
{
  let full = total / caseSize
  let remainder = total % caseSize
  return CaseSplit { full, remainder }
}
```

## 【critical】Opus 側で別途実行した `kei check --generative` が反例を検出

```text
$ cargo run -p kei_cli --bin kei -- check --generative numeric_utils.kei

error[KEI-E4005]: ensures 'result.remainder >= 0' of 'splitIntoCases'
  is violated by a generated input (total = -1, caseSize = 2)
  --> numeric_utils.kei:25:11
   |
25 |   ensures result.remainder >= 0
   |           ^^^^^^^^^^^^^^^^^^^^^
   = fix: Fix the implementation to satisfy the contract, or correct the contract
```

### 反例の解析

Kei の `/` は 0 方向切り捨て、`%` は同じ商の剰余:

- `-1 / 2 = 0`
- `-1 % 2 = -1`
- → `remainder = -1`、`remainder >= 0` 違反

サブエージェントが書き忘れた事前条件は `total >= 0`(または実装で `remainder` を補正)。

### 決定的な signal

1. **M23 generative は正しく機能している** — 7 ラウンド全部で「reasonable に見えるコード」を初回 PASS させてきたが、ここで初めて **生成的反例検出で破綻** を発見。
2. **Sonnet が「論理的に妥当に見える契約」を書いても negative input で破綻する** → Kei の generative 検証は **本当に価値がある**。
3. **しかし MCP `kei_check` 経由では `verification: "runtime"` 固定でこの事実が見えない** — R6 で発見した「MCP contracts が runtime 固定」の **重大性を裏付ける証拠**。

つまり Kei の「コンパイラが履行を保証する」設計前提のうち、**最も価値の高い部分(generative 反例検出)が、AI が主に触る MCP 経路では機能していない**。これは v0.4 ドッグフード全 7 ラウンドの **最重要発見** の 1 つ。

## 観測された摩擦(構築過程)

### 【最重要 1 / 剰余演算】手間ほぼゼロ

`§2 演算子表 + コメント「/ は 0 方向切り捨て、% は同じ商での剰余」`で即座に書けた。

### 【最重要 2 / 単項演算子】手間ゼロ

§2 演算子表に `単項 -x !x` と明記。`-n` も `n != 0` も迷わず書けた。

### 【最重要 3 / 複数契約】手間ほぼゼロ

「条件を『かつ』で並べたいときは `requires` / `ensures` を複数行に分ける(`&&` は無い)」が明示されていた。

### 取説になくて推測で書いた項目(medium 級)

**`ensures` の式で record 型の `result.フィールド` がアクセスできること** — `examples/contracts/counter.kei` は Int 戻り値の `ensures result == old(count) + step` のみで、record 戻り値の `result.full * caseSize + result.remainder == total` の例は無い。論理的には当然書けるが、明示サンプルがないため推測になった。
