## 重要な signals

### Positive(取説品質の証明)

1. **6 ラウンド連続で初回 PASS、SKILL.md 1 回読みのみ** — 関数 1 個(R1〜R3)から 4 ファイル分割(R5)、機能強制(R6)まで、SKILL.md Read 回数は常に 1。所要は規模で 90s → 319s と 3.5 倍になるが、ベース情報は SKILL.md 1 通読で揃う。
2. **「examples に同型例ゼロ」でも初見組み立て可能**(R3・R4・R5 で実証)— 取説の「原則 + 断片例 + エラーカタログ」構成が **同型コピペ不可能なドメインでも組み立て可能なレベル** に達している。
3. **`&&` 不在の太字明示が機能している** — 全ラウンドで「事前に気付いてハマらなかった」と報告。「**何が無いか**」を明示する書き方が効いている。
4. **MCP `kei_check` の source 直渡し設計が高評価** — 全ラウンドで「往復コストほぼゼロ」と評価。
5. **複数モジュール分割も初見で書ける(R5)** — `import module.path { Type1, Type2 }` 構文と「ファイルパスと module パスの 1:1 対応」が SKILL.md に明示されており迷わない。
6. **List リテラル / lambda 構文 / 契約構文は初見で書ける(R6)** — `[1, 5, 12, 24]` / `p => p > 0` / `ensures result.length <= prices.length` がそれぞれ SKILL.md の記述から直接書けた。`contracts` フィールドの自動返却も評価された。

### Negative(構造的に直すべき)

1. **【critical / R7 で確定 / R6 と統合】MCP `kei_check` で reasonable に見えるコードが generative 検証で破綻** — Sonnet が 7 ラウンド連続で MCP `kei_check` を通したが、R7 で Opus が CLI `kei check --generative` を別途回したら **反例検出**(`splitIntoCases(total=-1, caseSize=2)` で `result.remainder = -1` が `>= 0` 違反)。Kei の「コンパイラが履行を保証する」最も価値の高い部分(generative 反例検出)が、AI が主に触る MCP 経路では機能していない。
2. **【critical / R6 で発見】lambda キャプチャ禁止が contains 相当の超基本パターンを破綻させる** — 「外側変数を使う述語」=「contains 相当」=「最も基本的な業務ロジック」が書けない。KEI-E2001 の fix 候補(fold シードに渡す)も現実に通らない。`List.contains(item)` 相当の組み込みメソッドも無いため代替手段ゼロ。
3. **【critical / R5 で発見】MCP `kei_check` の import 解決が opaque か検査済か区別不能** — 「コンパイラが履行を保証する」の前提が MCP 経由で見えなくなる。M20 が実装された ≠ MCP 経由で実証可能。
4. **【R4/R5 で 2 回再現】本体 Bool 式での AND 表現イディオム不在** — 取説で最大のイディオム漏れと確定(2 ラウンド連続)。
5. **【high / R6 で発見】「キャプチャ禁止で何が書けなくなるか」のサンプル不足** — SKILL.md の「キャプチャ禁止」記載 ≠ 実用上の重大制約の説明。anti-example が必須。
6. **【high / R5 で発見】record の差分更新構文がない** — 規模が増えるほど深刻化。
7. **lambda(M25)が R1〜R5 連続で named function に流れる** — R6 で初めて踏まれたが、ここでもキャプチャ制約で重大な問題が顕在化。
8. **MCP `kei_check` の I/F が SKILL.md に未記載**(R1)— CLI 形式しか書いていない。
9. **【R7 で発見】負数入力の事前条件設計をガイドする例不足** — 数量保存契約で `total >= 0` を加えないと負数で破綻する落とし穴が、SKILL.md 例(counter.kei は `step > 0`)からは明示的に伝わらない。
10. **記法揺れ・例不足が散在**(R2: `length` / `List<Int>` / R3: `None()` vs `E.V` / `Ok(record リテラル)` / R4: 複数値返却 wrapper / `result` 衝突 / R5: String `==` / import 連鎖 / match 腕の式制約 / R6: List.contains 不在 / 未使用パラメータ silent / R7: `result.field` アクセス例不足)
