## 観測された摩擦

### 【critical】MCP `kei_check` の import 解決が機能しているか判別不能

サブエージェント原文:

> SKILL.md は「ファイルが見つからない場合は opaque(検査素通り)」と書いており、MCP の `kei_check` はソーステキストを渡す API なのでファイルシステムを参照できない。つまり今回の `import tickets.types { Ticket, Status, ... }` は実質 opaque 扱いで通過している可能性が高い。実際に型定義の解決が行われているか(M20 の cross-file 検証が効いているか)を確認する手段がツール上なく、「チェックが通った」理由が「正しいから」なのか「opaque 素通りだから」なのかが区別できない。

これは Kei の核となる設計前提「**コンパイラが履行を保証する**」が、MCP 経由ユーザーから **見えなくなっている** 重大な問題。M20 が「実装された」のと「MCP 経由でも実証可能」は別物。

### 【high】record の差分更新構文がない

6 フィールドの `Ticket` を 1 つだけ変えて返す関数で全フィールド列挙が必要。サブエージェント原文:

> `Ticket` は 6 フィールドあるため `resolve_ticket`・`close_ticket`・assign 内での更新は `ticket.id, ticket.subject, ticket.created_at, ...` と全部書く必要があった。スプレッド構文的なものがあれば楽だったが、仕様に言及なく、ないものとして全書きした。

assignment.kei では `Ok(Assignment { ticket: Ticket { id: ticket.id, subject: ticket.subject, ... }, agent: Agent { ... } })` が長大な 1 行になり可読性も低下。規模が増えるほど深刻化する構造的問題。

### 【medium】import した型のフィールドの型も import 必要かのルール未明示

`Assignment` が `Ticket` と `Agent` をフィールドに持つため、`assignment.kei` では `import tickets.types { Ticket, Agent, Assignment, Status, TicketError }` と全部明示する必要があった。「import した record のフィールドに使われている型まで明示 import が必要か」が事前不明で、試行錯誤で発見。

### 【medium】純粋 Bool 関数本体の AND 表現が R4 に続き R5 でも再現

`is_sla_violated` で `if is_urgent { if is_unresolved { return elapsed > 3600 } return false } return false` と入れ子 if で AND を表現。R4 の `overlaps` と同じパターン。**2 ラウンド連続で発生 = 取説で最大のイディオム漏れと確定**。

### 【medium】String の `==` 比較・`agent_id: -1` リテラル・`result.isOk` の使い方が未明示

- `subject == ""` は比較演算子表に `==` があり「型的に OK だろう」と推測で書いた(通った)
- `agent_id: -1` のリテラルが record フィールドに直書きできるか単項 `-` 説明から推測(通った)
- `result.isOk` を `ensures` で使う方法は今回未使用だが確証なし

### match の腕は式のみ(文ブロック不可)— 規模で再露出

> SKILL.md §5 末尾に「腕の本体は式のみ(文ブロック不可)」とあるが、小さく書かれており見落としやすい。assign_ticket の match 腕に複数の let 束縛を書きたかったが制約に気づいてワンライナー表現に切り替えた。

R3 でも触れられたが、規模が増えると match の使用機会が増え再露出。

## 助かった点(原文要約)

- 「書いたら必ず検証する」が最初に来る構成 → check ループに迷いなく入れた
- `match` の網羅性エラー(KEI-E2007)の事前説明 → 最初から全バリアントカバーした
- `else fail` の説明が豊富
- `kei_fmt` がフォーマット差分をゼロにする → 認知コスト低下
