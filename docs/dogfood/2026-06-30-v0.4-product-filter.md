### サブエージェントが書いた .kei

```kei
module products.filter

func allowedCategories() -> List<Int> {
  return [1, 5, 12, 24]
}

func positivePrices(prices: List<Int>) -> List<Int>
  ensures result.length <= prices.length
{
  return prices.filter(p => p > 0)
}

func isAllowedCategory(category: Int, allowed: List<Int>) -> Bool {
  // allowed パラメータは完全未使用 — キャプチャ禁止で汎用実装を断念
  if category == 1 { return true }
  if category == 5 { return true }
  if category == 12 { return true }
  if category == 24 { return true }
  return false
}
```

## 観測された摩擦

### 【critical】lambda キャプチャ禁止が「contains 相当」の超基本パターンを破綻させる

サブエージェント原文:

> `isAllowedCategory` で `c => c == category` を書いたところ、`category` が外側スコープのキャプチャに当たるとして `KEI-E2001` が出た。
> ...
> fix の提案 2 は「fold のシードに category を渡す」だったが、実際に試みると `fold(false, (found, c) => found || c == category)` でも同じキャプチャ問題が起きる。また `fold(category, ...)` では返り値型が `Int` になり `Bool` として返せない。**結局 `allowed` パラメータを活用した汎用実装は諦め、ハードコードに切り替えた。**

これは **言語設計上の重大な問題**:

1. 「**外側変数を使う述語**」=「**contains 相当**」=「最も基本的な業務ロジック」が書けない
2. **fix 候補(`fold` シードに渡す)が現実に通らない** → KEI-E2001 の fix 生成ロジックがバグに近い
3. `List.contains(item)` 相当の組み込みメソッドが無いため代替手段ゼロ
4. サブエージェントは「`allowed` パラメータ完全無視」で回避 → **仕様の理想と真逆の解**

### 【high】「キャプチャ禁止で何が書けなくなるか」のサンプルが SKILL.md に無い

サブエージェント原文:

> SKILL.md がこの制限を「キャプチャ禁止」と書いていても、「だから何が書けなくなるか」のサンプルが無いため、実際に壁に当たるまで認識しにくい。

「キャプチャ禁止・純粋限定」の 1 行 ≠ 実用上の重大制約の説明。anti-example が必須。

### 【observation / 設計検討】MCP `kei_check` の contracts は `verification: "runtime"`

ensures は書けて contracts に反映されたが、検証レベルは `runtime`(generative ではない)。M23 の本領である **生成的反例検出が MCP 経由ではデフォルト無効** に見える。これは `kei check --generative` フラグの設計と関係し、MCP ツール経由でも generative を呼べる経路が必要かもしれない。

### 未使用パラメータが silent に通る

`isAllowedCategory(category: Int, allowed: List<Int>)` で `allowed` が完全未使用なのに warning なし。Kei の設計判断としても要検討(unused-arg を warning する/しないは方針次第)。

### `List.contains(item)` 相当の組み込みメソッド不在

サブエージェントが探したが MCP `kei_spec` の `List contains` 検索が失敗。コンビネータ 8 種に `contains` は無し。lambda キャプチャ禁止と相まって、リスト検索が組み込みで書けない構造的問題。
