# kei fmt 正規形スタイル v0.1(M2)

> `kei_fmt` の出力規則。tests/golden/fmt/ が契約本文であり、本書はその要約。
> 規則の変更は golden の変更を伴うため人間レビュー必須(ARCHITECTURE.md 不変条件 3)。

## 全体

- インデントは半角スペース 2。タブ禁止。
- 1 行の最大幅は設けない(v0.1)。文・宣言ヘッダは折り返さない。
- ファイル末尾は改行 1 つ。空モジュールは空文字列。
- セクション(module 宣言 / import 群 / 各トップレベル項目)の間は空行 1 つ。
  import 同士は空行なしで連続させる。ブロック内に空行は置かない。
- import の並び順は **ソース順を保持**する(並べ替えは AST の意味的変更にあたるため
  v0.1 ではしない)。

## 宣言

- `module a.b` — 1 行。
- `import a.b { X, Y }` / `import a.b as Z` / `import a.b` — 1 行。
  名前列挙が空の `{ }` は素の import に正規化される。
- `type N = T tagged "tag"` — 1 行。
- record / enum — フィールド・バリアントを 1 行 1 件、カンマなし。
  バリアントのペイロードはインライン(`V(A, B)` / `V { a: A, b: B }`)。
  本体が空なら `record N {}` の 1 行。

## 関数

- シグネチャ(`func name(params) -> Ret`)は 1 行。
- 契約節(uses / requires / ensures)は各 1 行、インデント 2。
  uses は 1 行にカンマ区切り、requires / ensures は節ごとに 1 行。
- 契約節があるとき body の `{` は独立行(列 0)。ないときはシグネチャ行末に置く。
- 空ブロックは `{}`。

## 文・式

- 1 文 1 行。文は改行で区切る(セミコロンなし)。
- `else if` はチェーン(`} else if cond {`)。
- 括弧は構文上必要な最小限のみ:
  - 優先順位(弱→強): `implies`(右結合)< 比較 < `+ -` < `* /` < 単項 < 後置(`.` 呼び出し)。
  - if 条件の直下に露出する record リテラルは `( )` で包む
    (パーサの no_struct 文脈と対応)。
- 文字列は `\" \\ \n \t \r` のみエスケープし、他の文字はそのまま出力する。

## コメント(`//` 行コメント、M19 / v0.4)

- 整形でコメントは保持される。レキサーが `Comment` トークンとして
  捨てずに採取し、フォーマッタが行位置から leading / trailing を再構築する。
- **leading コメント**(別行で次の宣言・文より前にあるもの)は対象ノードと
  同じインデント段で 1 行ずつ出力する。連続コメントは詰めて 1 ブロック。
- **trailing コメント**(同じ行の末尾)はコード末尾と半角スペース 1 つ
  で区切って付け直す(`requires x > 0 // ...` のように 1 スペース)。
- **本体コメント**(関数本体ブロック内の独立行)はブロックのインデント段に
  揃えて出力する。最終文と `}` の間に残るコメントは `}` の直前にまとめる。
- ファイル末尾のコメントはセクション区切り(空行)を挟んで残す。
- 取りこぼし: 引数並びや式の中間に置かれたコメントは v0.4 では
  「次のアンカーノードの leading」に寄せて保持する(完全な位置復元ではないが
  テキストは失われない)。完全な位置保持は今後の拡張で対応。
