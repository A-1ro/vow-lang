# plan-then-delegate — Opus 計画 → Sonnet 実装の二段委譲

判断は Opus、コーディングの手数は Sonnet。Opus は**計画書を詰めるところまで責任を持ち**、実装は self-contained なプロンプトとして Sonnet サブエージェントに渡す。Opus の context を節約しつつ Sonnet の速さを活かす分業。サブエージェントは会話履歴を見ていない前提なので、計画書だけで動けるよう書く。

## 使う / 使わない

**原則として、コード編集を伴う実装タスクなら常に使う。** 以下はあくまで代表例で、これ以外でも該当すれば発火する:

- レビューコメントの一括対応(複数ファイルの機械的修正)
- /goal や Milestone で、修正箇所と方針が確定している実装
- リファクタ・命名変更・型シグネチャ変更の伝播(対象箇所を grep で確定できる)
- バグ修正・機能追加・テスト追加(修正対象がファイル:行レベルで特定できれば 1 ファイルでも対象)

**使わない**(Opus 自身でやる) — 本質的に二段委譲できないケースだけ:
- 設計判断が次々入る探索的なタスク(計画書を固定できない)
- `tests/golden/` の expected 改変・spec と実装の乖離判断が頻発する作業(不変条件の判断は Opus 同席が安全)

「1 ファイル数行だから委譲オーバーヘッドが大きい」という理由では除外しない。修正対象が確定していれば委譲する。

## 計画段階(Opus 担当)

1. **関連コードを読み、修正対象をファイル:行レベルで特定する。** 「だいたい emit.rs のあたり」では駄目。レビューコメントが指している実体を grep / Read で確かめる。
2. **各タスクを以下の粒度で書く**:
   - 修正対象ファイルと行番号
   - 現状(短い引用 or 要約)
   - 期待する修正の具体形(コードスニペット or 差分)
   - 根拠(レビュアー指摘・仕様参照・なぜそう直すか)
3. **検証コマンドを順番付きで列挙する。** Kei なら `kei-verify` skill 相当(`cargo fmt --check` → `cargo clippy --workspace --all-targets -- -D warnings` → `cargo test --workspace`)。
4. **コミット粒度を決める。** 1 コミットにまとめるか分けるか。`git commit` までやらせて push は人間が確認してから、が安全。
5. **「止まるべきトリガー」を明示する。** golden test 改変・spec 矛盾・想定外の差分・レビュアー指摘との解釈ズレ — これらは勝手に進めず相談、と書く。
6. **報告フォーマットを指定する。** 変更ファイル・行・検証コマンド結果を簡潔に返してもらう。

## 移譲段階

Agent ツールを以下で起動:

```text
subagent_type: claude
model: sonnet
prompt: <計画書(self-contained)>
```

プロンプトには **会話履歴の前提を入れない**。サブエージェントは「今このプロジェクトに初めて入った同僚」として読むので、リポジトリ名・絶対パス・なぜこれをやるかも含める。

## 検証段階(Opus 担当)

サブエージェントの完了報告は **「やったつもり」を述べるテキスト** であって、実際の diff ではない。最低限こうやって裏取りする:

- `git log -1 --stat` でコミットの存在と規模を確認
- `git show HEAD` または `git diff HEAD~1 HEAD -- <file>` で実差分を spot-check(報告された行と一致するか)
- 検証コマンド(fmt / clippy / test)が pass したかを自分でも 1〜2 個再現するか、log 末尾を読む

不一致や違和感があれば、push 前に修正を指示する。push を任せると最後の砦が消える。

特に <一番踏みやすい不変条件 — 例: golden test の expected を実装都合で書き換えない> を厳守してください。

## このリポジトリ(Kei)での追加事項

- **検証コマンドの定型**: `cargo fmt --all -- --check` → `cargo clippy --workspace --all-targets -- -D warnings` → `cargo test --workspace`(`kei-verify` skill と同じ)。サブエージェントには生コマンドで渡す(スキル名だけ渡しても辿れない)。
- **`tests/golden/` の expected** を実装都合で書き換えるのは CLAUDE.md の不変条件 1 違反。落ちたら止まって相談、と必ず明示する。**ただし** `crates/*/tests/*.rs` の `assert!` で書かれた単体テスト(emit.rs のスナップショット等)は golden ではないので、実装の正当な変更に伴って書き換えてよい — 判断が分かれそうなら計画書で「これは golden ではない」と前置きする。
- **e2e の `package-lock.json`**(`tests/cli/projects/app/` と `tests/e2e/`)はテスト実行で変化することがある。サブエージェントには「コミット前に `git checkout --` で復元」を明示する。
- **依存方向**(`kei_syntax ← kei_check / kei_fmt ← kei_emit / kei_cli ← kei_mcp`)の逆流や、`Diagnostic` の他クレートへの漏洩が疑われる変更を依頼するときは、計画段階で **`kei-invariant-auditor` サブエージェントに先に診てもらう**ことを検討する。
- **spec / 実装の乖離**を起こす変更(構文追加・契約意味論変更など)は、計画段階で「spec を先に直すか実装を先に直すか」を Opus が判断してから渡す。Sonnet にどちらを先か任せると CLAUDE.md の Source of Truth ルールを踏む可能性がある。

## 実例

直近では PR#69 のレビューコメント 5 件(`%` の IIFE 撤回、`||` の表セルエスケープ ×2、到達不能 arm 削除、`Prec::Implication` への揃え)を本パターンで処理。Opus が各コメントを ファイル:行 + 修正具体形 + 根拠 まで詰めた計画書を渡し、Sonnet が 1 コミットにまとめて検証まで通した。完了後 Opus が `git show HEAD --stat` と `git diff HEAD~1 HEAD` で報告内容を裏取りしてから push。
