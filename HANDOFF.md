# Kei — 設計判断ハンドオフ (HANDOFF.md)

> このファイルは、Kei 言語の**設計判断の経緯(なぜ今こうなっているか)**を
> Claude Code に引き継ぐためのもの。
> 「どう書くか(文法・イディオム)」は SKILL.md / spec/ を見ること。
> ここに書くのは「なぜその設計を選んだか」と「踏んだ地雷」。

## このプロジェクトは何か(最上位定義)

> **Kei の出発点は intent-align(AIと人間が実装前に意図を合意する)だった。**
> **意図を言語に落とし込むには DDD(ドメインを型と境界で表現する)が必要だった。**
> **結果として、TDD(契約からテストが導出される)を実現する言語になった。**
>
> **目的は intent-align、手段は DDD、結果は TDD。**
> 機能で迷ったら「これは意図の合意を強くするか?」で判断する。針は常に intent-align を指す。

この因果が Kei の憲法。以下の具体的な立場(検証層・立場B・捏造不能テスト等)は、すべてこの一点から導かれる枝である。

AI が第一の書き手であることを前提に設計したプログラミング言語。
人間の書きやすさを捨て、**検証可能性・推論の局所性・エージェントループとの親和性**に全振りする。
TypeScript にトランスパイルして V8 / Cloudflare Workers / Node で動かす実務言語。

**一行で:** 「AIと人間の合意書としてのコード」を成立させる言語。Code is a 契.

**立ち位置:** エージェントハーネスの **「検証層」**(verifiable IR for agent output)。
オーケストレーション(サブエージェント管理・ワークフロー制御)はやらない。それは Claude Code / CrewAI 等の領分。
Kei はハーネスの競合ではなく、ハーネスが生成する成果物を検証可能にする最後のピース。
迷ったら「これは検証層を強くするか?」も副次の判断軸になる(最上位は intent-align)。

## 譲れない設計原則(これを壊す変更はしない)

1. **合意書原則** — シグネチャ + 契約(`uses`/`requires`/`ensures`)だけ読めば、
   人間は body を読まずに承認判断できる。レビュー単位は実装ではなく契約。
2. **曖昧さゼロ** — 暗黙の型変換・暗黙の import・演算子オーバーロードなし。
   トークン数より推論コストを優先(AIにとって書くコストは安い、曖昧さは高い)。
3. **推論の局所性** — 1ファイルの挙動はそのファイル+明示importだけで決まる。
   再エクスポート禁止、グローバル可変状態なし。
4. **正規形唯一** — 同じASTは常に同じテキストに整形(`kei fmt`)。
5. **エラーは構造化データ** — Diagnostic は JSON が正、散文は派生。
   生成→検証→修正ループの帯域を最大化する。これが言語の存在意義。
6. **null 不在** — Option/Result のみ。例外機構なし。失敗は型に現れる。

## なぜ Rust 実装か

- コンパイラの実装言語と出力ランタイムは別物。Rust製コンパイラでも出力TSは
  Workersで動く(esbuild=Go製, SWC/Biome=Rust製と同じ構図)。
- 将来コンパイラ自体をWASM化してエッジ配布する道が開ける。
- パーサ系エコシステムが強い(logos, chumsky, ariadne)。

## なぜ MCP "だけ" では不十分か(重要な方針転換)

- 当初は「MCPサーバーを取説にして学習データ不足を推論で埋める」構想だった。
- **実測で MCP は遅かった**(図書館システムのドッグフードで体感)。
  理由: 文法を引く・検証する・修正するたびにラウンドトリップが累積する。
- 結論: **静的な知識(文法・イディオム・エラー対処)は Skill で一括ロード、
  動的な検証(kei check)だけ CLI/MCP で実行**、と役割分担する。
- MCP は捨てない。Skill を持たない任意のMCPクライアント向けの汎用口として残す。
- 優先度: **Skill を先に充実させる**(実体験が「Skillの方が速い」と言っている)。

## なぜ `kei mcp` を足したか(配布の一本化)

- 問題: MCP サーバーを使うたび利用者が手元で `cargo build -p kei_mcp` を回すのは負担。
  更新のたびにローカルビルドが要る運用は続かない。
- 方針(intent-align): 配布物は **`kei` 1 バイナリに統合**する。CLI を入れた人がそのまま
  `kei mcp` で取説サーバーを起動でき、Claude Code には `claude mcp add kei-mcp -- kei mcp` と
  配布バイナリを指すだけ。更新は `kei` の入れ直し(install.sh / `cargo install --force`)で済み、
  ローカルビルドが消える。
- 実装: `kei_mcp` の stdio ループを lib の `run_stdio` に切り出し、`kei-mcp` バイナリと `kei mcp`
  サブコマンドが**同じエントリを共有**する(プロトコルの二重実装を作らない)。
- 依存契約: `kei_cli → kei_mcp` を新設。一方向で循環しない(kei_mcp は kei_cli を知らない)。
  `kei_cli` は MCP プロトコルを持たず、起動を委譲するだけ — 「kei_cli は言語処理ロジックを
  持たない」原則は保たれる。`kei-mcp` 単体バイナリは開発・後方互換で残す。
- 踏まなかった代替: ① `kei mcp` から `kei-mcp` を別プロセス spawn → 2 バイナリ配布が残り統合に
  ならない。② MCP を廃して Skill のみ → 任意の MCP クライアント向けの汎用口を失う。どちらも却下。

## 命名の経緯(同じ轍を踏まないために)

- 当初名 "Pact" → Kadena Pact(ブロックチェーン言語)と同名・同思想で検索全滅 → 却下。
- 次に "Vow" → **Paulo Matos の vow-lang(形式検証エージェント言語)が完全先行**。
  ドメイン vow-lang.com・インストーラ・Claude Code skill まで揃った本職プロジェクト。
  思想までほぼ一致(収斂進化)。同名で戦うのは消耗なので撤退。
- 現名 **"Kei" = 「契」の音読み**。名前空間チェック済み:
  - crates.io `kei` / npm `kei` 単体はスクワッターが確保済み(中身ゼロ)
    → コンパイラは `keic`、クレートは `kei_*` prefix、npm は `@kei/*` スコープで回避。
  - ドメイン `kei-lang.dev` は取得可能。
  - 同音の "Kai"(LLVM系低レベル言語)が存在するが綴り違いで直接衝突なし。
- **教訓: 名前は惚れる前に crates.io / npm / ドメイン / 検索順位を必ず確認する。**

## 作業の流儀

- spec/ が source of truth。仕様と実装が食い違ったら**仕様を先に直す**。
- golden test の expected を実装都合で書き換えるのは契約の一方的変更。人間レビュー必須。
  (この警戒は `kei test` の「捏造不能性」原則=オラクルは契約のみ、に構造として昇華した。)
- Milestone 完了ごとに kei fmt を全コードに適用し、spec との乖離は spec を先に直す。
- **スコープ発散が新言語の第二の死因**。一体ずつ育てる(P1→P2→test→collection)。「全部入り」は「何も完成しない」と同義。
