# KEI-E2010: equality is not supported on this type

型検査: `==` / `!=` をスカラー以外の型に適用した。v0.1〜v0.3 の等値比較は
**スカラー限定**(`Int` / `String` / `Bool` と、それらを基底にする tagged 型)。

合成型(`Record` / `enum` / `Option<T>` / `Result<T, E>` / `List<T>` / `Unit`)に
対する `==` は許さない。トランスパイル先の TypeScript では `===`(参照等価)に
なり、**構造等価にならない**ためである。例えば `List` の `===` は同一オブジェクト
参照のときだけ真になるので、`ensures result == xs.get(0)` のような自然な契約が
非空リストでは**常に偽**になり、契約が常に失敗してしまう。サイレントな
不一致を避けるため、コンパイル時に弾く。

```kei
func sameList(a: List<Int>, b: List<Int>) -> Bool {
  return a == b   // error: equality is not supported on 'List<Int>'
}
```

## 修正

- スカラーのフィールド同士を比較する(例: `a.length() == b.length()`)。
- 値の中身を調べたいときは `match` で開く(`Option` / `Result` / enum)。
- 構造等価そのものが必要なら、目的に応じた述語関数を自分で定義する。

## enum / `Option` も一律 `match` で書く(設計判断)

ペイロードの無い enum バリアントはトランスパイル先でシングルトン定数
(`{ kind: "Foo" } as E`)として 1 度だけ生成されるため、原理的には `===` でも
正しく比較できる。それでも本エラーは **enum を一律で弾く**。等値可能な型を
「全バリアントがペイロード無しの enum」に限定すると、どの値が `==` できるかが
ソースから読み取りにくくなるためである。`Option` / `Result` / enum は **`match` で
開く**方針に固定する(可読性と将来の網羅検査の観点でも筋が通る)。

## 破壊的変更(v0.3)

KEI-E2010 は v0.3 で追加された。それ以前は合成型同士の `==` / `!=` が型適合さえ
すればコンパイルを通り、実行時に参照等価で**黙って誤動作**していた。`enum` /
`Option<T>` / `Result<T, E>` / レコードを `==` で比較していたコードは、移行時に
新たに KEI-E2010 が出る。上記の `match`・述語関数・スカラー比較へ書き換える。

構造等価(structural equality)の言語サポートは将来課題。導入時はこの制限を
緩める(spec/kei-spec-v0.3-collections.md の「等値比較」節を参照)。
